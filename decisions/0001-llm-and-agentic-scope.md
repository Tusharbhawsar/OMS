# ADR 0001 — LLM Usage and Agentic Scope for the Outage Communication System
## 1. Summary (TL;DR)

| Question | Decision | One-line rationale |
| --- | --- | --- |
| Use an LLM for **core logic** (who/when/how to notify)? | **No** | Deterministic rules are auditable, cheaper, faster, and compliance-safe. |
| Use an LLM for **message wording** (personalization + multilingual)? | **Yes — scoped** | Adds real value for multilingual reach and tone, with a deterministic fallback. |
| Make the workflow **agentic** (LLM autonomously decides + uses tools in a loop)? | **No (Phase 1)** | The workflow is predictable and compliance-critical; autonomy adds risk, not value. |
| Revisit agentic design later? | **Conditional** | Only for open-ended problems such as **unplanned/emergency** outage triage. |

**Net position:** Keep all business-critical decisions deterministic. Use the LLM strictly as a
**message writer**, never as a decision-maker. This is intentionally **not** an agentic system.

---

## 2. Context

The system ingests planned-outage data, identifies affected customers (via service-point /
circuit mappings), and dispatches multi-channel (SMS / Email) notifications across the outage
lifecycle (Advance Notice, Reminder, Outage Start, Restored, Cancellation), prioritising
**Medical Baseline** customers. Timing and eligibility are governed by fixed business rules and a
scheduled batch job.

This is a **regulated, customer-facing communication path**. The two qualities that matter most
are **correctness** (the right customer is told the right thing at the right time) and
**auditability** (every decision can be explained after the fact). Those qualities frame every
decision below.

---

## 3. Decision 1 — No LLM in core logic

**Decision:** Outage validation, customer mapping, notification timing, and lifecycle status
transitions remain 100% deterministic (SQL + explicit business rules).

**Why:**

- **Auditability / compliance.** "Why did customer X get message Y at time Z?" must have a
  deterministic, reproducible answer. Rules give that; a model does not.
- **Correctness.** Mapping and timing are exact problems with known-correct answers. An LLM can
  only introduce error here, never improve accuracy.
- **Cost & latency.** A batch can touch many customers; per-decision model calls would add cost
  and delay with zero upside.
- **Reliability.** No external dependency in the critical path — the notification engine works
  even if the LLM provider is down.

---

## 4. Decision 2 — LLM for message wording (scoped, with fallback)

**Decision:** Message **text** is generated by Google Gemini (Vertex AI) — personalised (e.g. an
added urgency line for Medical Baseline customers) and written in the customer's
`preferred_language` (currently English / Spanish). A deterministic template service is the
fallback.

**Why this is justified:**

- **Multilingual reach** without maintaining a combinatorial explosion of hand-written templates
  as languages grow.
- **Tone personalization** (urgent vs. reassuring) without writing a separate template per
  segment per language.
- **Future flexibility** for richer, situation-specific content that is hard to pre-template.

**Honest scoping note:** At the current scale (2 languages × 5 message types), fixed templates
would also be sufficient and are strictly more predictable. The LLM is therefore a
**strategic / scalability choice**, not a hard technical requirement. Its value grows with the
number of languages and the degree of personalization required.

**Guardrails that keep it safe:**

- The LLM only **writes**; it makes **no** routing, timing, or eligibility decisions.
- **Fact-preservation guardrail:** generated text must contain the exact date/time facts; if not,
  it is rejected.
- **Deterministic fallback:** if the LLM is disabled (`LLM_ENABLED=false`), errors, or fails the
  guardrail, the system uses fixed templates — notifications never break.
- **Low temperature** for fact fidelity in customer-facing messaging.

---

## 5. Decision 3 — Not an agentic system (Phase 1)

**Decision:** The workflow is a fixed, linear pipeline
(`validate → identify customers → dispatch`). There is **no** autonomous, tool-calling,
reasoning-in-a-loop agent. (LangGraph is used only as a lightweight step orchestrator.)

**Why agentic would be the wrong choice here:**

| Agentic property | Effect on this system |
| --- | --- |
| Non-determinism | Same input could yield different actions — unacceptable for compliance. |
| Tool-calling autonomy | Risk of the model notifying the wrong customers or skipping notifications. |
| Reason-act loop | Adds latency and cost to a high-volume batch with no decision ambiguity. |
| Harder to audit | Weakens the explainability the regulated path depends on. |

For a **planned-outage** workflow with **known rules**, deterministic code is not just adequate —
it is the *correct* engineering choice. Autonomy solves problems this system does not have.

---

## 6. Alternatives considered

1. **Fully deterministic, no LLM at all** (templates only).
   Viable today; chosen *against* only to gain multilingual scalability and personalization.
   Remains the automatic fallback, so we retain its reliability.
2. **LLM in the decision path / fully agentic.**
   Rejected: introduces non-determinism, cost, latency, and audit risk into a compliance-critical
   flow, with no corresponding benefit for a rules-based problem.
3. **Current choice — deterministic core + LLM-as-writer with fallback.**
   Accepted: keeps correctness/auditability intact while enabling multilingual, personalized
   messaging and a clean upgrade path.

---

## 7. Consequences

**Positive**

- Critical decisions stay deterministic, auditable, and reliable.
- Multilingual + personalized messaging with a safe degradation path.
- The LLM can be turned off instantly (`LLM_ENABLED=false`) with no functional loss beyond wording.

**Trade-offs / risks**

- An external dependency (Vertex AI) and per-message cost — bounded to the *messaging* layer only.
- Model/quota/region availability must be monitored (e.g. the project is currently entitled to
  `gemini-2.5-flash`).
- Translation quality for added languages needs review before go-live.

---

## 8. When to revisit

Re-open the **agentic** question only if the scope expands to genuinely open-ended decisions, e.g.:

- **Unplanned / emergency** outages with ambiguous or incomplete data.
- ETR estimation, dynamic escalation paths, or coordination across field crews / weather /
  customer-complaint signals.
- Decision spaces too large to express as explicit rules.

Re-open the **LLM-for-messaging** scope if: many more languages are required, or message content
becomes too dynamic to template — strengthening the case already made above.
